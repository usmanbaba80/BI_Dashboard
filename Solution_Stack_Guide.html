The Big Picture — What Is This All About?

This document is your client's complete blueprint for a data pipeline. Before diving into individual tools, let's understand what the whole system does in plain English — no jargon, no acronyms.

🏭
The Factory Analogy

Imagine a factory that turns raw materials into finished products. Raw data arrives from 7 different suppliers (Firebase, GA4, AdMob, AppLovin, Unity, Meta Ads, TikTok). The factory processes it through three assembly lines (Bronze → Silver → Gold), and the finished product is a beautiful set of business dashboards that tell the CEO exactly how the company is performing — every single morning.

🌍 Who Is The Client?

A Melbourne-based small/medium business (SMB) that develops mobile and gaming apps. They publish apps across multiple platforms and make money through advertising (AdMob, AppLovin) and paid user acquisition (Meta Ads, TikTok Ads). Currently, they have no unified view of their data — they log into 7 different portals and manually copy numbers into spreadsheets. This pipeline fixes that.

Infrastructure

Contabo VPS — Your Rented Servers

Before any software runs, you need computers to run it on. Instead of buying physical servers or using expensive managed clouds, the plan uses Contabo — a budget European hosting provider with a Sydney datacenter — to rent two virtual private servers (VPS).

🖥️
Plain English: What is a VPS?

A VPS is like renting a desk and computer in a giant office building (the datacenter). The building handles electricity, cooling, and internet. You get your own dedicated space with guaranteed resources. You can install any software you want, run it 24/7, and access it remotely from your laptop via SSH (a secure terminal connection).

🖥️ Two Servers, Two Jobs
⚡

Runs all the pipeline software: Airbyte, Dagster, dbt-ui, Ollama
6 CPU cores 12GB RAM 100GB SSD Sydney
This server is the "brain" of the system. It runs a tool called Docker Compose, which lets you run multiple software applications side-by-side as isolated containers. Think of Docker containers like separate rooms in the same building — Airbyte lives in room 8000, Dagster in room 3000, dbt-ui in room 3001, and Ollama in room 11434. (The "room numbers" are port numbers.)
You access these apps from your browser: http://SERVER_IP:3000 opens Dagster, http://SERVER_IP:8000 opens Airbyte, etc.
🗄️
DB VPS — Cloud VPS 30
Dedicated database server — stores all your data
8 CPU cores 24GB RAM 400GB NVMe SSD Sydney
This server's only job is to run PostgreSQL — the database. Databases are performance-sensitive and need dedicated resources. Putting it on a separate server means the database isn't competing with Airbyte and Dagster for CPU and RAM.
The two servers communicate over an internal connection on port 5432 (the PostgreSQL port). The App VPS writes data to the DB VPS, and Power BI on your laptop reads data from the DB VPS.
300GB of the 400GB disk is allocated to the analytics database — enough to store approximately 2–3 years of daily data at current volumes.
🔐 Security Setup
⚠️ Security is your first job

Before installing any software, you configure firewalls (UFW on Linux) so that the database port (5432) only accepts connections from the App VPS's IP address — not the entire internet. This is critical. An unsecured PostgreSQL port is one of the most common ways servers get compromised.

Ingestion Layer

Airbyte — The Data Collector

Airbyte is an open-source tool that knows how to talk to hundreds of different data sources (APIs, databases, files) and copy their data into your database automatically. It's the first step in the pipeline — getting data from the outside world into your system.

🚚
Plain English: What Airbyte does

Imagine you have 7 suppliers who all speak different languages and send invoices in different formats. Airbyte is a universal translator and delivery driver. It knows how to speak to Google, Meta, TikTok, Firebase etc. in their own language (their API), grab the data you need, convert it into a consistent format, and drop it in your database — every day, automatically.

⚙️ How Airbyte Works — Step by Step
1
You configure a "Connection" in the UI

Open Airbyte at http://SERVER_IP:8000 in your browser. Click "New Connection", choose the source (e.g. Google Analytics 4), enter your credentials (API key, account ID), choose Postgres as the destination, and set a sync schedule.

2
Airbyte calls the source API

At the scheduled time, Airbyte makes authenticated HTTP requests to (e.g.) Google's API asking for yesterday's analytics data. It handles API pagination, rate limits, and authentication automatically — you don't write this code.

3
Incremental sync — only new data

Airbyte is configured for "incremental" mode, meaning it tracks a watermark (like a date or ID) and only pulls new rows since the last sync. This keeps it fast and avoids re-pulling the entire history every day.

4
Data lands in Postgres staging tables

Airbyte writes the raw JSON/CSV data into PostgreSQL staging tables like raw_ga4_stg, raw_firebase_stg, etc. This is the Bronze layer — raw and untouched.

5
Dagster monitors the sync status

Airbyte reports success/failure back to Dagster. If a sync fails, Dagster raises an alert and stops the downstream pipeline steps from running on incomplete data.

Orchestration Layer

Dagster — The Pipeline Conductor

Dagster is the "brain" that sequences every step in the right order at the right time. It says: "At 2 AM, start Airbyte. When Airbyte finishes, run dbt Silver. When that's done, run dbt Gold. Then run quality checks. Then refresh Power BI." If anything fails, it stops and alerts you.

🎼
Plain English: What Dagster does

Dagster is like a stage manager for a theatre production. The actors (Airbyte, dbt, Power BI) all know their lines, but they need someone to tell them when to go on stage, in what order, and to stop the show if an actor doesn't show up. Dagster also keeps a history of every performance — so you can look back and see exactly what happened at 2:35 AM last Tuesday.

🧩 Key Dagster Concepts
◆
Assets

In Dagster, everything is an "asset" — a piece of data that gets created or updated. An Airbyte sync is an asset. A dbt model (e.g. silver.ga4_events) is an asset. A Power BI dashboard is an asset. Dagster tracks the state of every asset and shows you a visual graph of how they all connect.

◆
Asset Graph (DAG)

Dagster draws a visual diagram called a DAG (Directed Acyclic Graph) showing which assets depend on which. You can see: GA4 raw data → Silver GA4 table → Gold app metrics → Power BI dashboard. If you change the GA4 model, Dagster shows you which downstream dashboards will be affected.

◆
Schedules & Sensors

Schedules trigger at fixed times (e.g. "run the nightly batch at 2 AM"). Sensors watch for events (e.g. "if new rows appear in raw_applovin_stg, trigger dbt"). Together they cover both time-based and event-driven pipelines.

◆
Integrations with Airbyte and dbt

Dagster has official plugins (dagster-airbyte and dagster-dbt) that automatically import Airbyte connections and dbt models as Dagster assets. You don't manually wire them — Dagster discovers them automatically.

📝 What You Write in Dagster

Dagster is configured in Python. The main file is definitions.py:

Python — dagster definitions.py (simplified)
from dagster import Definitions, ScheduleDefinition, AssetSelection
from dagster_airbyte import load_assets_from_airbyte_instance
from dagster_dbt import DbtCliResource, dbt_assets

# Auto-load ALL Airbyte connections as Dagster assets
airbyte_assets = load_assets_from_airbyte_instance(
    airbyte_instance,
    asset_key_prefix=["airbyte"]
)

# Auto-load ALL dbt models as Dagster assets
@dbt_assets(manifest=dbt_manifest)
def mobile_analytics_dbt_assets(context, dbt: DbtCliResource):
    yield from dbt.cli(["run"], context=context).stream()

# Nightly batch schedule: run everything at 2 AM
nightly_schedule = ScheduleDefinition(
    name="nightly_batch",
    cron_schedule="0 2 * * *",  # 2:00 AM every day
    target=AssetSelection.all()
)

defs = Definitions(
    assets=[airbyte_assets, mobile_analytics_dbt_assets],
    schedules=[nightly_schedule],
)

Transformation Layer

dbt Core + dbt-ui — The Data Transformer

dbt (Data Build Tool) takes the raw messy data that Airbyte dropped in Postgres and transforms it into clean, business-ready tables using SQL. It's your main engineering tool — this is where the most analytical thinking happens and where you'll write the most code.

👨‍🍳
Plain English: What dbt does

dbt is the chef in the kitchen. Airbyte delivered the raw ingredients (vegetables, meat, spices — all unprocessed). dbt takes those ingredients and follows recipes (SQL files) to cook them into finished dishes. Each SQL file is one recipe, and the finished dishes are the clean tables that Power BI serves to the business.

🏗️ How dbt Works
1
You write SQL files called "models"

Each SQL file is a model — a SELECT query that defines a table. The file name becomes the table name. silver/stg_ga4_events.sql creates the silver.stg_ga4_events table.

2
Models reference other models

You use {{ ref('model_name') }} to reference other dbt models. dbt automatically figures out the build order — it knows Silver must be built before Gold, without you needing to specify that manually.

3
dbt compiles and runs the SQL

When you run dbt run, it takes your SQL files, adds CREATE TABLE / INSERT logic around them, and executes them on PostgreSQL in the correct dependency order. All in seconds.

4
dbt test checks the output

dbt has a built-in testing framework. You write tests in YAML files: "revenue should never be negative", "user_id should never be null", "store should be one of these 5 values". Run dbt test and it runs all assertions automatically.

Warehouse Layer

PostgreSQL + TimescaleDB + Citus — The Data Warehouse

PostgreSQL is where all the data lives. It's the world's most popular open-source relational database. Two extensions supercharge it for this use case: TimescaleDB makes it fast for time-series data (which app analytics always is), and Citus allows it to scale across multiple servers when data grows.

🏦
Plain English: Three schemas, three purposes

The database is like a bank with three vaults. The raw_* staging vault holds the unprocessed cash Airbyte deposits. The silver vault holds sorted, counted, verified bills. The gold vault holds the final investment portfolio that Power BI reports on. Each vault has strict access rules — and you only move money forward, never backward.

⚡ TimescaleDB — Why It Matters

App analytics data is almost always time-series — every row has a timestamp. TimescaleDB is a PostgreSQL extension that automatically partitions tables by time into "chunks" (called hypertables). The benefit: querying "show me last 30 days" only reads last month's chunk, not 3 years of data. Queries become 10–100x faster. It also automatically compresses old data by 90%+ to save disk space.

SQL — Setting up a TimescaleDB hypertable
-- Create the table normally first
CREATE TABLE silver.ga4_events (
    event_timestamp  TIMESTAMPTZ NOT NULL,
    user_pseudo_id   TEXT,
    event_name       TEXT,
    platform         TEXT,
    revenue          DECIMAL(10,4)
);

-- Turn it into a TimescaleDB hypertable (partitioned by time)
SELECT create_hypertable('silver.ga4_events', 'event_timestamp');

-- Automatically compress chunks older than 7 days (saves 90% disk)
SELECT add_compression_policy('silver.ga4_events', INTERVAL '7 days');

-- Auto-delete data older than 90 days from raw staging
SELECT add_retention_policy('silver.ga4_events', INTERVAL '90 days');
🌐 Citus — Horizontal Scaling (Future)

Citus is a PostgreSQL extension that distributes tables across multiple servers ("shards") for even larger datasets. For this client it's installed but not heavily used yet — it becomes important when data grows beyond 50GB/day in Phase 3. Enabling it is a single SQL command:

SQL — Enabling Citus distribution
-- Shard the gold.app_metrics table by app_id across worker nodes
SELECT create_distributed_table('gold.app_metrics', 'app_id');

Operations

The Daily Pipeline — What Runs & When

Once built, the entire pipeline runs automatically every night without you touching anything. Here's exactly what happens from 2:00 AM to 3:20 AM every single day in Melbourne time.

🌙 Nightly Batch Pipeline (2:00 AM – 3:20 AM AEDT)
2:00
Airbyte — GA4 + Firebase Sync

Dagster fires the trigger. Airbyte pulls yesterday's GA4 events (~500MB) and Firebase app events (~300MB) from Google's APIs. Data lands in raw_ga4_stg and raw_firebase_stg.

2:10
Airbyte — AdMob + Meta + TikTok + AppLovin + Unity Sync

All 5 ad platforms synced in parallel (~200MB total). Ad revenue, spend, impressions all land in their respective staging tables.

2:30
Dagster — Validation Checks

Dagster verifies all Airbyte syncs completed successfully. Checks row counts, confirms schemas are correct. If anything failed, Slack alert fires immediately and downstream steps are skipped.

2:35
dbt — Silver Layer (~15 min)

Dagster triggers dbt run --select silver.*. dbt parses JSON from GA4, cleans nulls, deduplicates events, standardises timestamps to UTC, joins events with revenue. Creates clean Silver tables.

2:50
dbt — Gold Layer (~10 min)

Dagster triggers dbt run --select gold.*. Daily KPIs aggregated, LTV cohorts calculated, campaign ROI computed, retention funnels updated. Gold tables are now ready for Power BI.

3:00
Postgres — Maintenance

TimescaleDB compresses data older than 7 days. VACUUM ANALYZE runs on modified tables to reclaim disk space and update query planner statistics. This keeps queries fast as data grows.

3:05
Power BI — Dataset Refresh (~10 min)

Dagster calls the Power BI REST API to trigger a dataset refresh. Power BI imports fresh Gold data. All 5 dashboards updated. By 3:15 AM, dashboards show last night's data.

3:15
Ollama — AI Daily Summary

Python script queries gold.app_metrics and gold.campaign_roi. Sends data to Ollama. LLM generates plain-English summary. Posted to #data-insights Slack channel.

3:20
✅ Pipeline Complete

Dagster marks all assets as "materialised". Email report sent: total data processed, run duration, error count. Team wakes up at 9 AM to fresh dashboards and an AI summary in Slack.

⚡ Live 15-Minute Pipeline (24/7)

Running in parallel with (and independent from) the nightly batch, a separate Dagster job fires every 15 minutes around the clock. This keeps the Campaign ROI and Real-Time dashboards updated continuously during business hours:

Every 15 minutes: Meta API + GA4 Real-Time API → Python aggregation → Postgres rt_metrics → Power BI auto-refresh

This pipeline runs for approximately 2 minutes. It pulls the last 2 hours of campaign data, computes current-hour ROAS and CAC, and upserts the results into the gold.rt_metrics table. Power BI's Campaign ROI dashboard queries this table directly.
